# 猎聘/Boss直聘 自动化投递项目 — 复盘总结

> 项目周期：约 30+ 小时（含 WorkBuddy 先期探索 200+ 小时的经验继承）
> 最终成果：双平台自动化投递 + 健康值守 + 系统托盘 + 断点续传
> 核心教训：**90% 的浪费来自方案错误，不是代码错误**

---

## 一、走过的弯路（按时间线）

### 弯路 1：不读页面就写代码

**现象**：项目初期直接打开编辑器写 hover、click、sendResume 逻辑，完全凭"猜测"页面结构。

**后果**：
- 写了大量在真实页面上根本不工作的代码
- hover 触发用 CDP mouseMoved → 不生效 → 换 dispatchEvent → 不生效 → 来回换
- 确认弹窗选择器猜了 5-6 个版本

**根因**：没有先通过 DevTools / CDP evaluate 读取页面的真实 DOM 结构。

**正确做法**：
```
先读 10 分钟页面 → 再写 5 分钟代码
而不是
先写 30 分钟代码 → 再调试 2 小时
```

### 弯路 2：一个变量反复修，不换思路

**现象**：hover 触发不了聊一聊按钮，反复调整 mouseMove 坐标（right-30 → right-5 → right-8 → right-10）、等待时间（500ms → 800ms → 400ms）、步数（5步 → 3步 → 20步）。调了 7-8 次全失败。

**后果**：浪费了约 4 小时在同一个思路上。

**分水岭**：最终通过诊断脚本 `console.log('hover@('+px+','+py+') => ' + tag)` 发现 CDP mouseMoved **确实到达了正确位置**，但页面不渲染按钮。这才意识到是 **CDP 触发不了 React 合成事件**的问题，而不是坐标不对。

**正确做法**：每次调试带上诊断日志，先确认"问题到底是什么"，再决定改哪里。

### 弯路 3：过度架构 — 在需要简单循环时引入复杂引擎

**现象**：早期把 `_runBossPages` 设计成带有健康时钟、多模式切换、断点续传的三层引擎，但 Boss直聘的流程其实只有"点立即沟通→点留在此页"两步。

**后果**：代码复杂度远超实际需求，首次跑通时间被推迟。

**正确做法**：先让最简单的路径跑通，再逐步加保护层。**功能完整度 ≠ 代码复杂度**。

### 弯路 4：容忍"静默失败"

**现象**：`engine.evaluate()` 返回 `undefined` 或 `null` 时，代码只是跳过继续，没有输出任何日志。`getCardStatuses()` 返回空数组时，代码自动跳到下一个模式，用户看到页面"什么都没发生"。

**后果**：问题被掩盖，花了大量时间猜"为什么没动"。

**正确做法**：每个 evaluate 都应有 `console.log()` 输出结果，特别是失败路径。**可观测性是调试的基础**。

### 弯路 5：字符串拼接中的引号地狱

**现象**：在 `evaluate` 字符串里嵌套 `${cardSel}`，而 `cardSel` 本身包含双引号：
```javascript
var cardSel = 'a[data-nick="job-detail-job-info"]';
// 拼接后变成：
// document.querySelectorAll("a[data-nick="job-detail-job-info"]")
// 语法错误！
```

**后果**：登录检测循环一直失败，进程永远在"等待页面加载"，用户看到 Chrome 打开但什么都不做。

**修复**：改成 `\\\"` 转义内层引号。

**教训**：**模板字符串拼接 evaluate 代码时，一定要检查生成的 JS 语法是否正确**。可以先 `console.log` 出拼接结果，然后用 `node -e` 验证。

## 二、WorkBuddy 继承的关键教训

### WorkBuddy 最大的错误

**重写已工作的代码**。v8 版本已验证成功（269/68），但后续从 v9 到 v21 全部在"优化"已工作的部分，结果越改越差，最终 308 次误触，成功率降到 52。

**教训**：如果一段代码在生产跑通了，**不要为了"优雅"重写它**。只修补实际 bug，不做预防性重构。

### WorkBuddy 最大的正确决定

**停用 `__reactFiber$` 读取**。通过 React Fiber 直接操作组件状态看似强大，但实际上破坏了 React 的事件系统，导致 CDP 事件全面失效。

**教训**：**不要碰框架内部状态**。永远通过用户可见的 DOM 元素去交互，走正常的事件循环。

### 正确 vs 错误做法总结

| 做法 | WorkBuddy 结果 | 我们验证 |
|------|---------------|---------|
| React Fiber 操作 | ❌ 事件系统崩溃 | ✅ 从不使用 |
| 暴力清除蒙层 | ❌ 308 次误触 | ✅ 精确选择器 |
| 字符串匹配状态 | ❌ 误判 46 次 | ✅ JSON 序列化 |
| 轮询等抽屉 | ❌ 5-10s 卡顿 | ✅ sleep(2000) 简单可靠 |
| 全页面搜按钮 | ❌ 宽泛→误点 | ✅ scoped 到 dialog 内 |
| 多步重试 | ❌ 越试越乱 | ✅ withRetry + 明确超时 |

## 三、正确开发流程（可复用方法论）

### 3.1 诊断优先（替代"猜测驱动开发"）

遇到任何一个 SPA 自动化问题，先做三件事：

```
① evaluate 读 DOM 结构（卡片、按钮、弹窗的 class 和 text）
② evaluate 读 iframe 结构（有无跨域限制）
③ 手动在 DevTools 验证选择器
```

这步应该花 **10-15 分钟**，写代码之前必须完成。

### 3.2 30 分钟原则

如果一个问题修了 **3 次还没好**（比如 hover 不生效、弹窗点不到），说明方案错了，**不是参数不对**。

此时应该：
```
① 停下来，不修第 4 次
② 问自己：这个方案真的可行吗？
③ 写诊断代码获取真实数据
④ 如果自己诊断不了 → 请求用户从 DevTools 协助
```

### 3.3 可观测性规范

每个关键操作必须输出：
```
操作名 + 位置/参数 + 结果
```

示例：
```
clickChat start @(611,329)   ← 输入：坐标
hover@(611,329) => A.t=职位名  ← 中间：鼠标位置上的元素
found btn @(752,403): "聊一聊" ← 输出：找到了什么
```

### 3.4 渐进式构建

```
第一阶段：让最简路径跑通
  → 点一个卡片 → 发简历 → 确认 → 打招呼 → 成功
  
第二阶段：加批处理
  → 遍历所有卡片 → 翻页 → 遍历关键词
  
第三阶段：加固
  → 重试机制 → 超时保护 → 风控降速
  
第四阶段：管理
  → 健康时钟 → 断点续传 → 托盘界面
```

**不要在第一阶段做第四阶段的事。**

### 3.5 字符串拼接防护

涉及 `evaluate(JS 字符串)` 时：

```javascript
// ❌ 错误：直接拼接变量
var sel = 'a[data-nick="job-detail-job-info"]';
engine.evaluate('document.querySelectorAll("' + sel + '").length');
// → "a[data-nick="job-detail-job-info"]" 语法错误！

// ✅ 正确：转义内层引号，或提前验证
var sel = 'a[data-nick=\\"job-detail-job-info\\"]';
// 或使用模板字符串 + 单引号包裹
```

**黄金法则**：拼好后 `console.log` 出来检查，或在 DevTools Console 里先跑一遍。

## 四、技术决策记录

### 4.1 为什么用 CDP 而不是 Puppeteer/Playwright？

| 因素 | CDP | Puppeteer |
|------|-----|-----------|
| 依赖 | 零（Node.js 内置） | 需安装 ≈50MB |
| 灵活性 | 完全控制 WebSocket | 封装好的 API |
| 稳定性 | 无版本冲突 | 需匹配 Chrome 版本 |
| 学习成本 | 需要了解 CDP 协议 | 更低 |

**结论**：零依赖 + 直接控制 WebSocket 更适合此类工具类项目。但团队项目建议用 Playwright。

### 4.2 为什么用 Electron 而不是原生托盘？

- 开发速度快（HTML/CSS/JS）
- 跨平台兼容（macOS/Windows/Linux）
- 社区成熟，文档丰富

**代价**：包体积大（~200MB），GPU 兼容性问题（Intel Mac + 新版 Electron）。

### 4.3 CSS 选择器策略

```
精确 class 选择器（优先） > contains 选择器 > 文本匹配
```

优先用 DevTools 复制来得精确 class，不要用 `[class*=xxx]` 模糊匹配。前者稳定，后者容易因为页面改版失效。
